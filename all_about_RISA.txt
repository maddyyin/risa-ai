1. PROJECT OVERVIEW
--------------------------------------------------------------------------------
   Name        : RISA - Intelligent Assistant
   Full Form   : Routine Intelligence & Scheduling Assistant
   Type        : AI-Powered Productivity & Life Automation Web Application
   Version     : 0.1.0 (Prototype)
   Description : RISA is a futuristic, AI-driven life automation platform that
                 helps users master their day through intelligent scheduling,
                 smart task management, behavior learning, and comprehensive
                 productivity analytics. It uses Google Gemini 2.5 Flash to
                 analyze user habits, generate productivity insights, and
                 intelligently auto-schedule tasks across time blocks. All
                 task data persists in a local PostgreSQL database via Prisma
                 ORM with an optimistic-UI approach and local storage fallback.

   Tagline     : "Master Your Day with Intelligent Automation"


2. OBJECTIVE / PURPOSE
--------------------------------------------------------------------------------
   The primary objective of RISA is to:

   a) Reduce Decision Fatigue
      - Automatically structure morning, afternoon, and evening time slots
      - Let AI handle the "when should I do this?" question for users

   b) Intelligent Task Scheduling
      - Provide AI-powered auto-scheduling (Gemini 2.5 Flash) that distributes
        tasks across time blocks (morning, afternoon, evening) based on
        estimated effort, priority, category, and configurable aggressiveness
      - Allow drag-and-drop manual scheduling with an interactive planner
      - Falls back to local heuristics when Gemini API is unavailable

   c) Behavior Learning & Adaptation
      - AI analyzes the user's complete task list and provides behavioral
        observations (e.g., "tendency to overbook evening blocks")
      - Identifies peak focus windows based on task completion patterns
      - Generates personalized productivity suggestions

   d) Boost Productivity
      - Track productivity scores and task completion rates dynamically
      - Visualize focus trends, weekly completion charts, and category
        breakdowns with interactive Recharts-powered visualizations
      - Offer AI-generated insights and reflections

   e) Persistent Data with Optimistic UI
      - All tasks persist in PostgreSQL via Prisma ORM through REST API routes
      - Optimistic UI updates for instant feedback; server sync in background
      - Local storage backup as a fallback when the API is unreachable
      - Soft real-time syncing via 12-second polling interval

4. TECHNOLOGY STACK
--------------------------------------------------------------------------------

   FRONTEND FRAMEWORK:
   - Next.js 16.2.3 (App Router architecture)
   - React 19.2.4
   - TypeScript 5.x

   STYLING:
   - Tailwind CSS 4.x (with PostCSS via @tailwindcss/postcss)
   - Shadcn UI (v4.2.0) - Component library
     (button, card, dialog, input, label, select, textarea)
   - Base UI (@base-ui/react ^1.3.0) - Low-level headless components
   - tw-animate-css 1.4.0 - Animation utilities
   - Dark theme by default (class="dark" on <html>)
   - Glassmorphism design with backdrop-blur effects
   - Gradient text effects (indigo → purple → pink palette)
   - Custom CSS design system in globals.css (glass-card, glow-button,
     gradient-text, task-card, sidebar-link, toggle-switch, etc.)

   STATE MANAGEMENT:
   - Zustand 5.0.12 (lightweight store for task state, CRUD, and polling)

   BACKEND / DATABASE:
   - PostgreSQL (local, on port 5432, database: risa_db)
   - Prisma ORM 5.22.0 (@prisma/client + prisma CLI)
   - Prisma schema: single "Task" model with UUID primary key
   - Next.js API Routes (App Router) for REST endpoints:
       * GET  /api/tasks          → Fetch all tasks
       * POST /api/tasks          → Create a new task
       * PUT  /api/tasks/[id]     → Update a task by ID
       * DELETE /api/tasks/[id]   → Delete a task by ID
       * GET  /api/ai/insights    → Generate AI productivity insights
       * POST /api/ai/schedule    → AI auto-schedule unassigned tasks
   - Singleton Prisma client pattern (dev hot-reload safe)

   AI / INTELLIGENCE:
   - Google Gemini 2.5 Flash (via @google/genai ^2.6.0)
   - Server-side API routes call Gemini with structured JSON output
     (responseMimeType: "application/json" + responseSchema)
   - /api/ai/insights: Gemini analyzes full task list from PostgreSQL and
     returns: score (0-100), peakHours, suggestion, atRisk[], and
     behaviorObservation
   - /api/ai/schedule: Gemini receives unassigned tasks + user settings
     (aggressiveness, workHoursLimit) and returns taskId→block mapping
   - Client-side lib/ai.ts wraps API calls with graceful fallbacks:
     * generateAIInsights() → calls /api/ai/insights, falls back to mock
     * autoBalanceSchedule() → calls /api/ai/schedule, falls back to local
       heuristic scheduler (least-load-first distribution algorithm)
   - Firebase SDK (^12.12.0) is still listed as a dependency but is NO LONGER
     ACTIVELY USED — the project has fully migrated to PostgreSQL/Prisma

   DRAG & DROP:
   - @hello-pangea/dnd 18.0.1 (fork of react-beautiful-dnd)

   ANIMATIONS:
   - Framer Motion 12.38.0 (imported but only used on landing page)
   - Custom CSS keyframe animations: fade-in-up, fade-in, slide-in-left,
     pulse-glow, counter-up, bar-grow, line-draw
   - Staggered delay classes: delay-100 through delay-800

   DATA VISUALIZATION:

     Used on the Analytics page
   - Custom SVG charts on the Dashboard page (bar chart + line/area chart)

   ICONS:

   FONTS:

6. DATA MODELS
--------------------------------------------------------------------------------

   Task (TypeScript Interface — src/types/index.ts):
   ├── id               : string              (UUID from Prisma)
   ├── title            : string              (Task name, required)
   ├── description      : string?             (Optional details)
   ├── status           : 'todo' | 'in-progress' | 'done'
   ├── estimatedMinutes : number              (Time estimate in minutes)
   ├── timeBlock        : 'morning' | 'afternoon' | 'evening' | undefined
   ├── priority         : 'low' | 'medium' | 'high' (optional, defaults medium)
   ├── category         : string?             (e.g. "Development", "Design")
   ├── dueDate          : string?             (ISO date string "YYYY-MM-DD")
   └── createdAt        : Date                (Auto-generated timestamp)

   Task (Prisma Schema — prisma/schema.prisma):
   ├── id               : String   @id @default(uuid())
   ├── title            : String
   ├── description      : String?
   ├── status           : String   @default("todo")
   ├── estimatedMinutes : Int      @default(30)
   ├── timeBlock        : String?
   ├── priority         : String   @default("medium")
   ├── category         : String   @default("General")
   ├── dueDate          : String?
   └── createdAt        : DateTime @default(now())

   DaySchedule (TypeScript Interface):
   ├── morning          : Task[]
   ├── afternoon        : Task[]
   ├── evening          : Task[]
   └── unassigned       : Task[]

9. DATABASE & BACKEND CONFIGURATION
--------------------------------------------------------------------------------

   PostgreSQL:
   - Host: localhost:5432
   - Database: risa_db
   - Schema: public
   - ORM: Prisma 5.22.0

   Required environment variables (.env or .env.local):

   DATABASE_URL="postgresql://postgres:<password>@localhost:5432/risa_db?schema=public"
   GEMINI_API_KEY="<your-gemini-api-key>"

   Firebase (legacy, still in .env.local but not actively used):
   NEXT_PUBLIC_FIREBASE_API_KEY=<key>
   NEXT_PUBLIC_FIREBASE_AUTH_DOMAIN=<domain>
   NEXT_PUBLIC_FIREBASE_PROJECT_ID=<project-id>
   NEXT_PUBLIC_FIREBASE_STORAGE_BUCKET=<bucket>
   NEXT_PUBLIC_FIREBASE_MESSAGING_SENDER_ID=<sender-id>
   NEXT_PUBLIC_FIREBASE_APP_ID=<app-id>

   Data Flow:
   1. StoreInitializer.tsx boots on app startup
   2. Zustand taskStore calls GET /api/tasks to fetch from PostgreSQL
   3. Tasks are cached in Zustand state + backed up to localStorage
   4. CRUD operations use optimistic UI (instant local update) then sync
      to PostgreSQL via API routes
   5. Polling every 12 seconds for soft real-time syncing
   6. If API fails, falls back to localStorage backup or mock data


10. HOW TO RUN
--------------------------------------------------------------------------------

    Prerequisites:
    - Node.js (v18+)
    - PostgreSQL running locally on port 5432
    - A Google Gemini API key (for AI features)

    Steps:
    1. Clone the repository
    2. Install dependencies:       npm install
    3. Create .env with:
         DATABASE_URL="postgresql://postgres:<password>@localhost:5432/risa_db?schema=public"
         GEMINI_API_KEY="<your-gemini-api-key>"
    4. Create the database:        createdb risa_db (or via pgAdmin)
    5. Run Prisma migration:       npx prisma migrate dev
    6. Generate Prisma client:     npx prisma generate
    7. Run the development server: npm run dev
    8. Open http://localhost:3000 in your browser

12. KNOWN ISSUES & TECHNICAL DEBT
================================================================================

    1. DEAD DEPENDENCY: Firebase SDK (firebase ^12.12.0) is listed in
       package.json and .env.local but no code imports or uses it. Should
       be removed to reduce bundle size.

    2. DUPLICATE ENV FILES: Both .env and .env.local contain DATABASE_URL
       and GEMINI_API_KEY with the same values. Should consolidate.

    3. NO AUTHENTICATION: The entire app is single-user with no login.
       All tasks are shared globally. The sidebar hardcodes "John Doe".

    4. NO TASK EDITING: Once a task is created, its title, description,
       priority, category, due date, etc. cannot be modified. Only status
       (todo/done) and timeBlock can be changed.

    5. NO "IN-PROGRESS" UI: The type system supports 'in-progress' status
       but no UI action sets it. Tasks can only toggle between todo↔done.

    6. SETTINGS DON'T PROPAGATE: Accent color, user name, email, and
       notification preferences are saved but never actually applied:
       - Accent color doesn't change the theme
       - User name doesn't update sidebar/header
       - Notifications don't exist

    7. ANALYTICS TIME RANGE IS FAKE: Week/Month/Year selector buttons
       don't change the data — always shows the same weekly view.

    8. CHART DATA ACCURACY: Charts derive data from createdAt day-of-week,
       not actual calendar date ranges. This means "weekly" data is not
       a real week view but a day-of-week aggregation across all time.

    9. MOBILE HAMBURGER MENU: The menu button renders but has no onClick
       handler or drawer/overlay implementation.

    10. SEARCH BAR (DASHBOARD): Accepts input visually but has no
        filtering logic connected to it.

    11. GEMINI API CALLS NOT CACHED: Every dashboard/analytics page load
        triggers a new Gemini API call. Should implement caching or
        debouncing.

    12. FRAMER MOTION UNDERUSED: framer-motion is installed (12.38.0) and
        imported on the landing page but only for the import — the actual
        motion components are not visibly used (animations are CSS-based).

    13. date-fns UNUSED: Listed as dependency (^4.1.0) but never imported
        in any source file.

    14. SIDEBAR AI BUTTON: The "AI Auto-Schedule" button in the sidebar
        is not connected to any logic.
